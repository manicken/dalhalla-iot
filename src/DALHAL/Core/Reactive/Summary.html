DALHAL Reactive Event System – Overview

The Reactive Event System in DALHAL provides a unified, type-safe, and flexible mechanism for devices to emit and subscribe to events without passing arbitrary values. It is designed to work efficiently with microcontroller memory and runtime constraints while supporting multiple devices, multiple event types, and multiple consumers.

Goals

  1. Automatic event emission: Devices automatically notify subscribers when specific conditions occur (e.g., value change, state change, write/read success, timeout, or custom events).

  2. Counter-based architecture: Each reactive event maintains a counter (uint32_t) representing the number of times the event has occurred. Subscribers track the last seen value to detect changes.

  3. Type safety and flexibility: Events are optionally implemented using a typesafe template-based table (EventDescriptorT<TDevice>) or a generic offset-based table (EventDescriptor).

  4. Minimal runtime overhead: All events are resolved via compile-time tables and macros. No runtime string parsing or dynamic registration is required.

  5. Ease of use for devices: Devices only need to declare which events they support via macros; the system generates tables, counters, and trigger functions automatically.

Core Concepts

1. Event Counters

  • Each device maintains counters for the events it supports, e.g., valueChange, stateChange, write, read, exec, timeout, custom.

  • Counters are incremented via trigger functions generated by macros:

inline void triggerValueChange() { reactiveEventCounter_ValueChange++; }

2. Event Tables

  • Each device declares a static event table listing the supported events:

#ifndef DALHAL_REACTIVE_NOT_USE_TYPESAFE_TEMPLATE_BASED_TABLES
static const EventDescriptorT<MyDevice> eventTable[];
#else
static const EventDescriptor eventTable[];
#endif

  • Entries map the event name to either a member pointer (uint32_t T::*) in typesafe mode, or a member offset (size_t) in generic mode.

  • Tables are terminated with a sentinel entry (nullptr), eliminating the need to store table size.

3. Event Retrieval

  • GetSimpleReactiveEventImpl: Core function to fetch a ReactiveEvent by name.

    • Typesafe mode: Uses uint32_t T::* member pointer.

    • Generic mode: Uses Device* pointer plus offset (offsetof) to locate the counter.

  • Returns a ReactiveEvent object, which wraps:

    • checkFn: Function to check if the event has occurred.

    • context: Pointer to the counter or context struct.

    • deleter: Function to clean up the context.

4. Reactive HAL Values

  • ReactiveHALValue wraps a normal HALValue but supports set and get callbacks, allowing automatic triggering of reactive events when the value changes or is read:

value.setCallbacks(ValueSetCallback, ValueGetCallback);

  • This allows the system to react immediately when device values are updated or read.

5. Event Features

The system defines granular event types as bitmask flags:

#define DALHAL_REACTIVE_FEATURE_VALUE_CHANGE   0x00000001
#define DALHAL_REACTIVE_FEATURE_STATE_CHANGE   0x00000002
#define DALHAL_REACTIVE_FEATURE_WRITE          0x00000008
#define DALHAL_REACTIVE_FEATURE_READ           0x00000004
#define DALHAL_REACTIVE_FEATURE_EXEC           0x00000010
#define DALHAL_REACTIVE_FEATURE_TIMEOUT        0x00010000
#define DALHAL_REACTIVE_FEATURE_CUSTOM         0x80000000

  • Devices declare which features they support via DALHAL_REACTIVE_CFG_<DEVICE> flags.

  • Custom events are supported using DALHAL_REACTIVE_FEATURE_CUSTOM.

6. Macros

  • Macros provide consistent naming and simplify reactive setup:

DECLARE_REACTIVE_FEATURE(ValueChange)
REACTIVE_ENTRY_VALUE_CHANGE(MyDevice)
HAS_REACTIVE(MyDevice, VALUE_CHANGE)
triggerValueChange()

  • This ensures type-safe and uniform implementation across all devices.

Design Decisions

  1. Counters instead of values: Events do not carry arguments — only the occurrence count is tracked. Consumers can retrieve the last seen value from the device.

  2. Optional typesafe tables: Using uint32_t T::* ensures full compile-time type checking but is optional. Generic offset-based tables allow simpler implementation and reduce template instantiations.

  3. Uniform event object: ReactiveEvent always stores a checkFn and Context pointer, and optionally a deleter. Even if a simple context would suffice, keeping this layout allows uniform handling for custom events.

  4. Callbacks on set/get: Reactive HAL values can automatically trigger events on modification or access.

  5. Single-source of truth: The event table and macros define all supported events; no separate arrays or redundant tables are required, avoiding mismatches.

  6. Extensible: Future event types or device-specific events can be added by defining new flags, macros, and context types.

Usage Pattern

  1. Declare reactive features in the device header:

#if HAS_REACTIVE(MY_DEVICE, VALUE_CHANGE)
DECLARE_REACTIVE_FEATURE(ValueChange)
#endif

  1. Define event table in the reactive .cpp file:

const EventDescriptorT<MyDevice> MyDevice::eventTable[] = {
    REACTIVE_ENTRY_VALUE_CHANGE(MyDevice),
    REACTIVE_ENTRY_STATE_CHANGE(MyDevice),
    {nullptr, nullptr}  // terminator
};

  1. Use triggers inside device methods:

value = 42;
triggerValueChange();

  1. Consumers retrieve events by name:

ReactiveEvent* event;
device->Get_ReactiveEvent("value_change", &event);

✅ Summary:

The DALHAL Reactive Event System provides a flexible, macro-driven, type-safe (optional), and memory-efficient way to implement event-driven behavior for all device types. By relying on counters, compile-time tables, and macros, it avoids runtime overhead while remaining extensible and easy to maintain.
